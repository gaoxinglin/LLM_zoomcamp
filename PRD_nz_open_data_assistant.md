# NZ Regional Insights Assistant — 产品需求文档（PRD）

| 项目 | 内容 |
|---|---|
| 文档版本 | 1.0 |
| 日期 | 2026-07-16 |
| 项目类型 | RAG + 工具调用（Agent）Web 应用 |
| 首发范围 | Auckland 地区的人口、住房与交通开放数据 |
| 主要数据源 | data.govt.nz Catalogue API、CKAN DataStore、政府机构公开 CSV |
| 目标用户 | 居民、学生、研究者、记者、政策分析初学者 |

## 1. 产品概述

NZ Regional Insights Assistant 是一个面向普通用户的新西兰开放数据助手。用户不需要了解政府机构名称、数据集标题、字段结构或 API，即可使用自然语言发现 Auckland 人口、住房和交通相关的开放数据，查询其中的事实，并获得带来源、更新时间和限制说明的回答。

产品第一版聚焦 Auckland，不尝试覆盖 data.govt.nz 的全部主题。系统同时提供两类能力：

1. **数据集发现**：根据自然语言问题推荐相关数据集，并解释数据集的内容、发布机构、时间范围、格式和适用场景。
2. **数据事实查询**：对已接入的结构化数据执行受控查询，完成筛选、聚合和地区/年份比较，并根据查询结果生成带引用的回答。

## 2. 问题陈述

新西兰政府公开了大量数据，但普通用户在使用时面临以下困难：

- 不知道数据由哪个政府机构发布，也不知道应使用什么关键词搜索。
- 数据集描述、文件格式和字段命名不统一。
- 找到数据集后，仍需下载 CSV、理解字段并编写查询或分析代码。
- 搜索结果和 LLM 回答可能缺乏来源、更新时间及数据限制说明。
- 一个问题可能涉及多个数据集，用户难以判断哪个更适合。

本产品通过混合检索、重排、查询改写及安全的数据工具调用，降低从“提出问题”到“找到可信数据或得到可验证答案”的门槛。

## 3. 产品目标

### 3.1 核心目标

- 用户能够用自然语言找到 Auckland 人口、住房和交通相关数据集。
- 对首批接入的数据集，用户能够查询数值、趋势、排名和地区/年份比较。
- 每个事实性回答都能追溯到具体数据集或资源。
- 明确显示发布机构、数据更新时间和数据限制。
- 数据或证据不足时，系统明确说明无法回答，不编造结论。
- 建立可重复运行的摄取、检索评估、LLM 评估和监控流程。

### 3.2 课程项目目标

- 实现知识库与 LLM 相结合的检索流程。
- 比较 BM25、向量检索、混合检索和重排方案。
- 比较至少两种 LLM 回答方案并采用表现更好的方案。
- 提供可交互 Web UI。
- 提供自动化数据摄取脚本。
- 收集用户反馈并提供至少五项监控图表。
- 使用 Docker Compose 运行应用及依赖。
- 提供完整、可复现的 README、示例和版本锁定。

## 4. 非目标

第一版不包含：

- 覆盖新西兰所有地区、机构和数据主题。
- 对任意外部数据文件自动生成可靠分析。
- 预测未来人口、房价或交通情况。
- 提供法律、投资、购房或政府政策建议。
- 将产品描述为新西兰政府官方服务。
- 支持用户写入或修改政府数据。
- 执行 LLM 自由生成的任意 SQL。
- 将地图可视化作为首发必要功能。

## 5. 目标用户与核心场景

### 5.1 Auckland 居民

希望快速理解所在地区的人口、住房或交通情况，但不熟悉政府数据门户。

示例问题：

- “哪里可以找到 Auckland 各地区的人口数据？”
- “这个数据集最后一次更新是什么时候？”
- “有哪些 Auckland 公共交通相关的开放数据？”

### 5.2 学生和研究者

需要发现可下载、可引用的数据，并了解数据的发布机构、字段和覆盖范围。

示例问题：

- “我想研究人口增长和住房许可，有哪些数据可以用？”
- “哪些数据集提供 CSV 或 API？”
- “这个数据适合做年度趋势分析吗？”

### 5.3 记者和初级分析人员

需要快速找到数据并验证简单事实，但最终结论仍需人工复核。

示例问题：

- “2023 年哪个地区人口最多？”
- “比较两个地区在两个年份的指标变化。”
- “答案来自哪个资源，原始链接是什么？”

## 6. 产品范围

### 6.1 元数据知识库

从 data.govt.nz Catalogue API 摄取 Auckland 及以下主题的相关数据集元数据：

- Population and demographics
- Housing and building
- Transport and roads

首发目标为 500–2,000 条数据集元数据。每条检索文档至少包含：

- 数据集 ID、标题和描述
- 发布机构
- 标签与主题
- 创建及更新时间
- 许可证
- 资源格式
- 数据集页面链接
- 资源下载/API 链接
- 是否支持 CKAN DataStore

### 6.2 可查询数据集

首发从元数据知识库中选择约 10 个结构清晰、来源稳定、许可明确的数据集作为“受支持数据集”。优先选择 CKAN DataStore 资源，其次选择稳定的 CSV 资源。

数据集必须满足：

- 与 Auckland 人口、住房或交通直接相关。
- 有明确发布机构和许可信息。
- 字段能够支持至少三个真实用户问题。
- 可通过 API 或稳定下载链接重复获得。
- 记录更新时间和数据覆盖时间。
- 不包含个人敏感信息。

具体数据集清单在技术调研阶段确认，并记录到 `data/catalog.yml`；PRD 不预设未经验证的数据集名称。

## 7. 用户体验与主要流程

### 7.1 主界面

界面包含：

- 自然语言问题输入框
- 示例问题
- 回答正文
- 数据来源卡片
- 原始数据集链接
- 更新时间和发布机构
- “有帮助/没帮助”反馈按钮
- 可选文字反馈

### 7.2 数据集发现流程

1. 用户输入自然语言问题。
2. 系统将问题分类为“数据集发现”。
3. 系统生成检索查询和可选同义词。
4. 混合检索返回候选数据集。
5. Reranker 对候选结果重排。
6. LLM 基于候选元数据生成推荐理由。
7. 页面展示最多五个数据集及其来源信息。

### 7.3 数据事实查询流程

1. 用户提出数值、趋势、排名或比较问题。
2. 系统识别指标、维度、筛选条件和时间范围。
3. 系统在受支持数据集中选择资源。
4. 查询计划通过 Pydantic schema 校验。
5. 工具层将计划转换为只读查询。
6. 查询结果连同数据来源交给 LLM。
7. LLM 生成答案、简要解释、来源和限制说明。

### 7.4 无法回答流程

出现以下情况时不得猜测：

- 没有找到相关数据集。
- 数据集没有用户要求的字段或时间范围。
- 数据资源无法访问或已过期。
- 多个数据集定义冲突，且无法确定应使用哪一个。
- 查询返回空结果。

系统应说明失败原因，并建议更具体的关键词、可用年份或候选数据集。

## 8. 功能需求

### 8.1 P0：首发必须完成

| 编号 | 需求 | 验收条件 |
|---|---|---|
| FR-01 | 元数据自动摄取 | 一条命令可分页获取、清洗并索引目标数据集 |
| FR-02 | 数据集搜索 | 用户问题可返回最多五个相关数据集 |
| FR-03 | 混合检索 | 同时使用关键词与向量检索，并提供统一排序 |
| FR-04 | 数据集详情 | 展示机构、描述、格式、许可、更新时间和原始链接 |
| FR-05 | 带来源回答 | 每个事实性结论至少关联一个数据来源 |
| FR-06 | 结构化数据查询 | 至少 10 个受支持数据集可通过受控工具查询 |
| FR-07 | 安全查询 | 仅允许白名单资源、字段、操作和只读查询 |
| FR-08 | 无答案处理 | 证据不足时明确说明，不能生成无依据数值 |
| FR-09 | 用户反馈 | 用户可提交点赞、点踩和可选文字反馈 |
| FR-10 | 监控页面 | 至少展示五张运行与质量图表 |
| FR-11 | 可复现运行 | Docker Compose 可启动应用及必要依赖 |
| FR-12 | 自动化评估 | 可重复执行 retrieval 和 LLM evaluation |

### 8.2 P1：完成 P0 后实现

- Query rewriting：将口语、缩写和地区别名转换为更适合检索的表达。
- Document reranking：对混合检索结果进行相关性重排。
- 对话上下文：支持用户在上一问题基础上继续筛选年份、地区或指标。
- 结果表格：事实查询可以展示原始结果的简化表格。
- CSV 下载：允许下载当前查询结果，而不是整个原始数据集。
- 缓存：缓存元数据、schema 和高频查询结果。

### 8.3 P2：可选增强

- 地图展示 Auckland 各地区指标。
- 定时增量更新数据。
- 扩展到 Wellington、Canterbury 或全国数据。
- 自动检测字段和数据质量。
- 云端部署。

## 9. 查询工具规范

LLM 不直接输出可执行 SQL。LLM 只生成结构化查询计划，例如：

```json
{
  "dataset_id": "supported-dataset-id",
  "resource_id": "resource-uuid",
  "dimensions": ["region"],
  "metric": "population",
  "aggregation": "sum",
  "filters": {
    "year": 2023
  },
  "sort": "desc",
  "limit": 20
}
```

工具层必须执行：

- 校验数据集和资源是否位于白名单。
- 校验字段、聚合函数和筛选操作。
- 只允许读取操作。
- 限制最大返回行数和请求超时。
- 拒绝注释、多语句和修改性 SQL。
- 记录查询计划、执行结果数量、耗时和错误。

## 10. 检索设计

### 10.1 索引文档

每个数据集生成一条主文档；描述过长时，可以拆分为“数据集描述”和“资源描述”子文档。索引保留 dataset ID，确保结果可聚合回数据集级别。

### 10.2 候选方案

- BM25：标题、描述、标签、机构和格式字段加权。
- Vector search：对标准化后的元数据文本生成 embedding。
- Hybrid search：融合 BM25 和向量排名。
- Reranking：对 Top 20 候选重排，返回 Top 5。
- Query rewriting：补充 Auckland 地区别名、主题同义词和数据格式意图。

最终线上方案必须由离线评估结果决定，而不是预先假定混合检索一定最佳。

## 11. 回答生成要求

系统提示词要求模型：

- 只根据检索元数据或工具查询结果回答。
- 区分“数据集描述”和“从数据中计算的事实”。
- 对每个关键结论标注来源。
- 显示数据更新时间或覆盖年份。
- 不把相关性误写成因果关系。
- 不把本产品称为官方政府服务。
- 信息不足时明确说明未知。
- 对可能存在定义差异的数据给出简短限制说明。

建议回答结构：

1. 直接答案
2. 支持数据或推荐数据集
3. 来源与更新时间
4. 必要的限制说明

## 12. 评估计划

### 12.1 Retrieval evaluation

建立不少于 60 个真实问题的 ground truth：

- 30 个数据集发现问题
- 15 个格式、机构和时间范围问题
- 15 个易混淆或无答案问题

每个问题标注一个或多个相关 dataset ID。比较以下方案：

1. BM25
2. Vector search
3. Hybrid search
4. Hybrid + reranker
5. Query rewriting + hybrid + reranker

指标：

- Hit Rate@5
- Recall@5
- MRR
- NDCG@5

上线门槛：

- Hit Rate@5 不低于 0.85
- MRR 不低于 0.70
- 最终方案在主要指标上优于单一 BM25 或单一向量基线

### 12.2 LLM evaluation

建立不少于 30 个端到端回答样本，比较至少两种 prompt 或模型方案。

评估维度：

- 问题相关性
- 事实正确性
- Groundedness
- 引用正确性
- 无答案处理
- 限制说明质量

上线门槛：

- 引用覆盖率 100%
- 高严重度虚构数值为 0
- Groundedness 合格率不低于 90%

### 12.3 确定性数据查询测试

为受支持数据集建立不少于 20 个带预期结果的查询案例。数值允许根据指标设置明确容差；数据版本更新后必须重新生成或审核 expected results。

### 12.4 人工评审

邀请至少三名试用者，每人完成五个任务并提交反馈。记录：

- 是否找到所需数据
- 是否理解答案来源
- 是否信任结果
- 完成任务所需时间
- 最大困惑点

## 13. 监控与反馈

系统记录：

- 请求时间和会话匿名 ID
- 原始问题与问题类型
- 检索到的 dataset ID 和分数
- 工具调用、耗时、状态和返回行数
- 模型、Token 使用量和估算成本
- 总响应时间
- 是否返回无答案
- 用户点赞、点踩和文字反馈

Dashboard 至少包含：

1. 每日请求量
2. P50/P95 响应时间
3. Token 使用量或估算成本
4. 点赞/点踩比例
5. 无答案率
6. API/DataStore 错误率
7. 高频主题或发布机构

监控数据不得保存 API 密钥、个人身份信息或不必要的敏感内容。

## 14. 非功能需求

### 14.1 性能

- 数据集发现请求 P95 小于 8 秒。
- 包含 DataStore 工具调用的请求 P95 小于 15 秒。
- 默认最多返回五个数据集和二十行查询结果。

### 14.2 可靠性

- 外部 API 失败时返回明确错误和可重试提示。
- 对元数据与 schema 使用本地缓存。
- 摄取过程支持分页重试和断点恢复。
- 单个损坏资源不得导致整个摄取任务失败。

### 14.3 安全与隐私

- 密钥只通过环境变量注入。
- 仓库提供 `.env.example`，不提交真实密钥。
- 查询工具遵守白名单和只读约束。
- 用户反馈不要求姓名或邮箱。
- UI 明确声明产品不是政府官方服务。

### 14.4 可访问性

- 不只用颜色表达状态。
- 链接文字具有明确含义。
- 表格和图表提供文字说明。
- 首发界面使用英文；架构保留后续增加中文界面的可能性。

## 15. 技术架构建议

| 层级 | 建议技术 |
|---|---|
| UI | Streamlit |
| API | FastAPI（首版可与 Streamlit 合并，稳定后拆分） |
| 检索 | Elasticsearch：BM25 + dense vector |
| 运行数据 | PostgreSQL；本地开发可先用 SQLite |
| LLM | OpenAI 或 Groq，通过统一 provider 接口封装 |
| Embedding | 可配置 embedding provider，并记录模型版本 |
| 数据验证 | Pydantic |
| 依赖管理 | uv |
| 测试 | pytest |
| 容器化 | Docker Compose |

逻辑组件：

```text
data.govt.nz / CSV
        ↓
Ingestion pipeline
        ↓
Metadata index + supported dataset registry
        ↓
Query classifier and rewriter
        ↓
Hybrid retrieval and reranking
        ↓
Safe data-query tools
        ↓
Grounded answer generator
        ↓
Streamlit UI + feedback + monitoring
```

## 17. 里程碑

### M0：数据可行性验证（1–2 天）

- 调通 Catalogue API。
- 搜索三个目标主题的数据集。
- 验证至少 10 个候选资源的访问、许可、字段和更新时间。
- 决定首发受支持数据集清单。

退出条件：至少 10 个数据集满足第 6.2 节标准；否则调整主题或减少功能范围。

### M1：检索 MVP（3–4 天）

- 完成元数据摄取脚本。
- 建立 BM25 和向量基线。
- Streamlit 可搜索并展示数据集卡片。
- 完成第一批 30 个 ground-truth 问题。

### M2：检索优化与回答生成（3–4 天）

- 完成混合检索、query rewriting 和 reranking 实验。
- 选择最优检索方案。
- 生成带来源的回答。
- 扩充到至少 60 个检索评估问题。

### M3：数据查询 Agent（4–5 天）

- 建立 supported dataset registry。
- 实现结构化查询计划和安全校验。
- 接入 CKAN DataStore/CSV adapter。
- 完成至少 20 个确定性查询测试。

### M4：监控与产品化（3–4 天）

- 接入反馈和运行事件记录。
- 完成至少五张监控图表。
- 完成错误处理、缓存和核心测试。
- Docker Compose 一键运行。

### M5：提交准备（2–3 天）

- 完成 README、架构说明和评估报告。
- 添加 UI、监控和回答示例截图。
- 从干净环境验证安装和运行步骤。
- 录制简短演示视频。

## 18. 发布验收标准

只有同时满足以下条件，MVP 才视为完成：

- 用户能够搜索 Auckland 人口、住房和交通数据集。
- 元数据摄取可通过一条文档化命令重复执行。
- 至少 10 个经过验证的数据集支持结构化查询。
- 每个事实回答都显示可点击来源和数据时间信息。
- BM25、向量和混合检索完成对比评估，线上使用最佳方案。
- 至少两种 LLM 回答方案完成评估，线上使用最佳方案。
- 通过至少 20 个确定性数据查询测试。
- UI 能收集点赞、点踩和文字反馈。
- Dashboard 至少包含五张有效图表。
- 所有服务可以通过 Docker Compose 启动。
- README 包含问题、数据、架构、评估、配置、运行方式、示例和限制。

## 19. 风险与缓解措施

| 风险 | 影响 | 缓解措施 |
|---|---|---|
| 数据资源格式不统一 | 任意数据查询难以可靠执行 | 首版只对白名单数据集编写 adapter |
| 外部链接失效 | 查询和演示失败 | 缓存 metadata/schema，记录最后成功更新时间 |
| 数据定义冲突 | 回答可能误导 | 显示机构、定义和限制，不自动合并不兼容指标 |
| LLM 生成虚构数值 | 降低可信度 | 数值只能来自工具结果；增加确定性测试 |
| API 响应慢或不可用 | 用户体验差 | 超时、重试、缓存和明确降级提示 |
| 项目范围过大 | 无法按时完成 | 锁定 Auckland + 三主题 + 10 个查询数据集 |
| 数据更新导致测试变化 | 回归测试不稳定 | 固定数据版本或在更新后审核 expected values |
| 用户误认为官方服务 | 信任和合规风险 | UI 和 README 明确标注非官方项目 |

## 20. 待确认问题

以下问题不阻塞 M0，但应在数据调研后确定：

1. 三个主题中是否都能找到足够稳定、可重复查询的数据集？
2. 首版是否只支持英文问题，还是增加中英文查询改写？
3. 运行数据库最终使用 PostgreSQL 还是 SQLite？
4. Reranker 使用本地 cross-encoder 还是 LLM/API？
5. 部署是否作为项目 bonus 范围？
6. 是否需要在 MVP 中展示简单图表，还是只展示文本和表格？

## 21. 成功定义

项目成功不以“能回答所有开放数据问题”为标准，而以以下结果为标准：

> 对 Auckland 人口、住房和交通领域的一组明确问题，系统能够稳定地找到正确数据集；对受支持的数据资源，能够安全、正确地查询数据；所有答案均可验证、可引用、可复现。
